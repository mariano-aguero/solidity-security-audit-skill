# AI-Generated Code Security Patterns

Security patterns specific to AI-assisted ("vibe-coded") Solidity.
LLMs have predictable failure modes that differ from human mistakes.
When AI generation is detected, increase manual scrutiny depth.

See `audit-questions.md` for function-level audit questions.
See `vulnerability-taxonomy.md` for full vulnerability descriptions.
See `secure-patterns.md` for correct implementation patterns.
See `automated-detection.md` for regex detection patterns.

## 1. Detecting AI-Generated Code

### 1.1 Codebase-Level Red Flags

```bash
# Mass commits: large diffs with uniform structure
git log --stat | grep -E "^\s+[0-9]+ files changed"

# "Generated by" comments
grep -r "generated by\|AI generated\|copilot\|ChatGPT\|claude" src/ --include="*.sol" -i

# Inconsistent naming (AI mixes styles within same file)
grep -n "temp[0-9]\|_temp\|tempVar\|helperFunc\|utilFunc" src/ --include="*.sol" -r

# Empty catch blocks (AI error handling anti-pattern)
grep -n "catch {}" src/ --include="*.sol" -r
grep -n "catch (Error\|catch (Panic" src/ --include="*.sol" -r

# require() with no message (AI often omits)
grep -rn "require([^,)]*)" src/ --include="*.sol"

# Mixed OZ v4 and v5 imports in same repo (copy-paste from different sources)
grep -rn "from.*@openzeppelin/contracts/" src/ --include="*.sol"
```

### 1.2 Function-Level Red Flags

- `address(0)` checks applied inconsistently (some functions yes, adjacent ones no)
- `nonReentrant` on some external functions but not others with identical risk profile
- `_msgSender()` and `msg.sender` mixed in same contract
- `emit Event(...)` after external call instead of before
- Comments describing the intent of code that does not match the actual implementation

## 2. LLM-Specific Anti-Patterns

### 2.1 CEI Violations

The most common LLM failure: making the external call before updating state.
LLMs generate the "readable" order (read → transfer → update) which is CEI-inverted.

**AI-generated (vulnerable):**
```solidity
function withdraw(uint256 amount) external {
    require(balances[msg.sender] >= amount, "Insufficient balance");
    // LLM puts transfer first because it "reads naturally"
    (bool success,) = msg.sender.call{value: amount}("");
    require(success, "Transfer failed");
    balances[msg.sender] -= amount; // State update AFTER external call
}
```

**Secure (CEI):**
```solidity
function withdraw(uint256 amount) external nonReentrant {
    require(balances[msg.sender] >= amount, "Insufficient balance");
    balances[msg.sender] -= amount; // Effects first
    (bool success,) = msg.sender.call{value: amount}(""); // Interactions last
    require(success, "Transfer failed");
}
```

**Audit check:** For every function that modifies state AND calls external code,
verify the exact order: Checks → Effects → Interactions.

### 2.2 Incomplete Access Control

LLMs add `onlyOwner` to obvious admin functions but miss:
- `initializer` / `onlyInitializing` on upgradeable contracts
- `onlyProxy` when using UUPS
- Role checks on functions only reachable through privileged paths
- Admin parameter validation (setting critical addresses to `address(0)`)

**AI-generated (vulnerable):**
```solidity
// UUPS upgradeable — AI adds onlyOwner but forgets to disable initializers
contract VaultV2 is UUPSUpgradeable, OwnableUpgradeable {
    function initialize(address token_) external initializer {
        __Ownable_init(msg.sender);
        token = token_;
    }

    // Missing: implementation can be initialized directly → ownership takeover
    // Should add: constructor() { _disableInitializers(); }

    function _authorizeUpgrade(address newImpl) internal override onlyOwner {}
    // Note: _authorizeUpgrade is internal — not externally callable.
    // The real risk above is the missing _disableInitializers() in the constructor.
}
```

**Secure:**
```solidity
contract VaultV2 is UUPSUpgradeable, OwnableUpgradeable {
    /// @custom:oz-upgrades-unsafe-allow constructor
    constructor() {
        _disableInitializers(); // Prevents implementation initialization
    }

    function initialize(address token_) external initializer {
        __Ownable_init(msg.sender);
        require(token_ != address(0), "Zero address");
        token = token_;
    }

    function _authorizeUpgrade(address newImpl) internal override onlyOwner {
        require(newImpl != address(0), "Zero address");
    }
}
```

**Audit check:** For every upgradeable contract: `_disableInitializers()` in constructor,
`onlyOwner` or `onlyRole` access control on `_authorizeUpgrade`, zero-address checks on `initialize` params.

### 2.3 Hallucinated Interface Functions

LLMs sometimes call functions that do not exist in the actual interface, or use
function signatures that were renamed between versions.

**Common hallucinations:**
```solidity
// ERC20 — LLM calls safeTransfer without importing SafeERC20
token.safeTransfer(recipient, amount); // CompileError if IERC20, not ERC20Safe

// Chainlink — wrong function name
(,int256 price,,) = feed.getLastestRoundData(); // typo: correct is latestRoundData

// Uniswap V3 — missing required 5th parameter (bytes calldata data)
pool.swap(recipient, zeroForOne, amountSpecified, sqrtPriceLimitX96); // wrong sig — missing 5th param: bytes calldata data (swap callback)

// OZ v5 — renamed from v4
token.safeApprove(spender, amount); // removed in OZ v5; use forceApprove() instead
```

**Audit check:** For every external call, verify the function signature against the
actual deployed interface, not just against what "seems right."

### 2.4 Broken Reentrancy Guards

LLMs add reentrancy guards but frequently misapply them:

**Pattern A — Guard imported but modifier never used:**
```solidity
contract Vault is ReentrancyGuard {
    // Guard imported but nonReentrant modifier never applied
    function withdraw(uint256 amount) external {
        // Vulnerable: no nonReentrant modifier
        (bool ok,) = msg.sender.call{value: amount}("");
    }
}
```

**Pattern B — Guard on internal helper instead of external entry point:**
```solidity
// Placing nonReentrant on an internal function is fragile when multiple external entry points
// exist: a second unguarded external function sharing state is not protected.
function _processWithdrawal(uint256 amount) internal nonReentrant { ... }
function withdraw(uint256 amount) external { _processWithdrawal(amount); } // OK if only entry
function harvest(uint256 amount) external { _processWithdrawal(amount); } // also guarded via nonReentrant on internal — but new state-sharing functions added later may not be
// Prefer: put nonReentrant on every external entry point explicitly
```

**Pattern C — Custom guard with storage collision in upgradeable contract:**
```solidity
// AI copies a simple lock variable, forgetting ERC-7201 namespaced storage
bool private _locked; // Slot 0 — collides with implementation storage layout on upgrade; use ReentrancyGuardUpgradeable (ERC-7201 namespaced)
modifier nonReentrant() {
    require(!_locked, "Reentrant");
    _locked = true;
    _;
    _locked = false;
}
```

**Audit check:** Verify `nonReentrant` is on the `external` entry point, not just internal
helpers. For upgradeable contracts, verify the lock variable uses namespaced storage or
use OZ's `ReentrancyGuardUpgradeable`.

### 2.5 EIP-712 / Signature Errors

LLMs generate incomplete EIP-712 implementations missing replay protection:

**AI-generated (vulnerable — missing nonce + deadline):**
```solidity
bytes32 public constant PERMIT_TYPEHASH = keccak256(
    "Permit(address owner,address spender,uint256 value)"
    // Missing: uint256 nonce, uint256 deadline
);

function permit(address owner, address spender, uint256 value, bytes calldata sig) external {
    bytes32 structHash = keccak256(abi.encode(PERMIT_TYPEHASH, owner, spender, value));
    bytes32 digest = keccak256(abi.encodePacked("\x19\x01", DOMAIN_SEPARATOR, structHash));
    require(ECDSA.recover(digest, sig) == owner, "Invalid sig");
    // No nonce → unlimited replay of same signature
    // No deadline → signature valid forever
    // No chainId in domain separator → cross-chain replay
    _approve(owner, spender, value);
}
```

**Secure:**
```solidity
bytes32 public constant PERMIT_TYPEHASH = keccak256(
    "Permit(address owner,address spender,uint256 value,uint256 nonce,uint256 deadline)"
);

function permit(
    address owner,
    address spender,
    uint256 value,
    uint256 deadline,
    uint8 v,
    bytes32 r,
    bytes32 s
) external {
    require(block.timestamp <= deadline, "Expired");
    bytes32 structHash = keccak256(abi.encode(
        PERMIT_TYPEHASH, owner, spender, value, nonces[owner]++, deadline
    ));
    bytes32 digest = _hashTypedDataV4(structHash); // Includes chainId + address(this)
    require(ECDSA.recover(digest, v, r, s) == owner, "Invalid sig");
    _approve(owner, spender, value);
}
```

### 2.6 Copy-Paste Inheritance Bugs

LLMs copy patterns from one contract and paste into another, breaking override chains:

```solidity
// AI copies ERC721 transfer hook but uses wrong super call order in diamond inheritance
contract MyNFT is ERC721, ERC721Enumerable, Pausable {
    function _update(address to, uint256 tokenId, address auth)
        internal override(ERC721, ERC721Enumerable) returns (address)
    {
        // Missing: whenNotPaused modifier copied from a Pausable example
        return super._update(to, tokenId, auth);
        // Risk: in diamond inheritance, super resolves via C3 linearization.
        // AI often copies the override list without verifying the MRO is correct.
    }

    // AI forgets to override supportsInterface for multiple inheritance
    // function supportsInterface(...) — missing, causes incorrect ERC165 responses
}
```

**Audit check:**
- [ ] Every `override(A, B, ...)` list is complete and matches the actual inheritance graph
- [ ] `supportsInterface` is correctly overridden when multiple standards are inherited
- [ ] `super` calls flow through the full C3 linearization chain

## 3. Audit Checklist for AI-Assisted Contracts

Run this in addition to the standard Phase 3 checklist when AI generation is suspected.

### 3.1 Pre-Review

- [ ] Search git log for mass commits (>20 files changed in one commit with uniform structure)
- [ ] Check for "generated by", "copilot", "ChatGPT", or "AI" comments in source
- [ ] Verify OZ version consistency across all imports (v4 and v5 not mixed)
- [ ] Identify functions with no NatSpec (AI sometimes omits for "obvious" functions)

### 3.2 Per External Function

- [ ] CEI order confirmed: Checks → Effects → Interactions (no exceptions)
- [ ] `nonReentrant` on every `external` function that calls out to external addresses
- [ ] Access control modifier exists AND covers all edge cases (init, proxy, admin)
- [ ] Every external call signature verified against actual deployed interface
- [ ] `require()` statements have descriptive messages

### 3.3 Per Signature Validation

- [ ] Domain separator includes `chainId` and `address(this)`
- [ ] TypeHash includes `nonce` and `deadline`
- [ ] Nonce is incremented after use (not reusable)
- [ ] Deadline checked before signature recovery

### 3.4 Per Upgradeable Contract

- [ ] `constructor() { _disableInitializers(); }` present on implementation
- [ ] `initialize()` uses `initializer` modifier (exactly once across upgrade chain)
- [ ] State variables use namespaced storage (ERC-7201) or are in safe upgrade-compatible slots
- [ ] `_authorizeUpgrade()` has proper access control and zero-address guard

## 4. Detection Patterns

```bash
# Likely CEI violation: .call{} not immediately preceded by state update
# (heuristic — review manually for false positives)
grep -rn "\.call{" src/ --include="*.sol" -A2 | grep -E "\-=|\+="

# Empty catch blocks
grep -rn "catch\s*{" src/ --include="*.sol"

# require() without descriptive message
grep -rn 'require([^,)]*[^,]*)' src/ --include="*.sol" | grep -v '"'

# tx.origin used for auth (AI sometimes uses this instead of msg.sender)
grep -rn "tx\.origin" src/ --include="*.sol"

# Potential safeTransfer hallucination (IERC20 does not have safeTransfer)
grep -rn "\.safeTransfer\b" src/ --include="*.sol"

# EIP-712 TypeHash missing nonce
grep -rn "TYPEHASH" src/ --include="*.sol" -A5 | grep -v "nonce"

# Missing _disableInitializers in upgradeable constructor
grep -rn "UUPSUpgradeable\|Initializable" src/ --include="*.sol" -l | \
  xargs grep -L "_disableInitializers"
```
