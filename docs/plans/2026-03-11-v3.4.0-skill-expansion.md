## 2. LLM-Specific Anti-Patterns

### 2.1 CEI Violations

The most common LLM failure: making the external call before updating state.
LLMs generate the "readable" order (read → transfer → update) which is CEI-inverted.

**AI-generated (vulnerable):**
```solidity
function withdraw(uint256 amount) external {
    require(balances[msg.sender] >= amount, "Insufficient balance");
    // LLM puts transfer first because it "reads naturally"
    (bool success,) = msg.sender.call{value: amount}("");
    require(success, "Transfer failed");
    balances[msg.sender] -= amount; // State update AFTER external call
}
```

**Secure (CEI):**
```solidity
function withdraw(uint256 amount) external nonReentrant {
    require(balances[msg.sender] >= amount, "Insufficient balance");
    balances[msg.sender] -= amount; // Effects first
    (bool success,) = msg.sender.call{value: amount}(""); // Interactions last
    require(success, "Transfer failed");
}
```

**Audit check:** For every function that modifies state AND calls external code,
verify the exact order: Checks → Effects → Interactions.

### 2.2 Incomplete Access Control

LLMs add `onlyOwner` to obvious admin functions but miss:
- `initializer` / `onlyInitializing` on upgradeable contracts
- `onlyProxy` when using UUPS
- Role checks on functions only reachable through privileged paths
- Admin parameter validation (setting critical addresses to `address(0)`)

**AI-generated (vulnerable):**
```solidity
// UUPS upgradeable — AI adds onlyOwner but forgets onlyProxy
contract VaultV2 is UUPSUpgradeable, OwnableUpgradeable {
    function initialize(address token_) external initializer {
        __Ownable_init(msg.sender);
        token = token_;
    }

    // Missing onlyProxy: anyone can call this on the implementation directly
    function _authorizeUpgrade(address newImpl) internal override onlyOwner {}

    // Missing: implementation can be initialized directly → takeover
    // Should add: constructor() { _disableInitializers(); }
}
```

**Secure:**
```solidity
contract VaultV2 is UUPSUpgradeable, OwnableUpgradeable {
    /// @custom:oz-upgrades-unsafe-allow constructor
    constructor() {
        _disableInitializers(); // Prevents implementation initialization
    }

    function initialize(address token_) external initializer {
        __Ownable_init(msg.sender);
        require(token_ != address(0), "Zero address");
        token = token_;
    }

    function _authorizeUpgrade(address newImpl) internal override onlyOwner {
        require(newImpl != address(0), "Zero address");
    }
}
```

**Audit check:** For every upgradeable contract: `_disableInitializers()` in constructor,
`onlyProxy` or equivalent on `_authorizeUpgrade`, zero-address checks on `initialize` params.

### 2.3 Hallucinated Interface Functions

LLMs sometimes call functions that don't exist in the actual interface, or use
function signatures that were renamed between versions.

**Common hallucinations:**
```solidity
// ERC20 — LLM calls safeTransfer without importing SafeERC20
token.safeTransfer(recipient, amount); // CompileError if IERC20, not ERC20Safe

// Chainlink — wrong function name
(,int256 price,,) = feed.getLastestRoundData(); // typo: latestRoundData

// Uniswap V3 — stale API (V2 syntax on V3 interface)
pool.swap(recipient, zeroForOne, amountSpecified, sqrtPriceLimitX96); // wrong sig

// OZ v5 — renamed from v4
token.safeIncreaseAllowance(spender, amount); // removed in v5, use forceApprove
```

**Audit check:** For every external call, verify the function signature against the
actual deployed interface, not just against what "seems right."

### 2.4 Broken Reentrancy Guards

LLMs add reentrancy guards but frequently misapply them:

**Pattern A — Guard exists but not applied:**
```solidity
contract Vault is ReentrancyGuard {
    // Guard imported but nonReentrant modifier never used
    function withdraw(uint256 amount) external {
        // Vulnerable: no nonReentrant modifier
        (bool ok,) = msg.sender.call{value: amount}("");
    }
}
```

**Pattern B — Guard on wrong function (internal instead of external):**
```solidity
// OZ ReentrancyGuard only protects within a single tx via _status
// Adding nonReentrant to internal function doesn't prevent cross-function reentrancy
function _processWithdrawal(uint256 amount) internal nonReentrant { ... }
function withdraw(uint256 amount) external { _processWithdrawal(amount); } // unguarded entry
```

**Pattern C — Custom guard with storage collision in upgradeable contract:**
```solidity
// AI copies simple guard, forgets ERC-7201 storage for upgradeable context
bool private _locked; // Slot 0 — collides with proxy's admin slot
modifier nonReentrant() {
    require(!_locked);
    _locked = true;
    _;
    _locked = false;
}
```

**Audit check:** Verify `nonReentrant` is on the `external` entry point (not just internal helpers).
For upgradeable contracts, verify the lock variable uses namespaced storage.

### 2.5 EIP-712 / Signature Errors

LLMs generate incomplete EIP-712 implementations missing replay protection:

**AI-generated (vulnerable — missing nonce + chainId):**
```solidity
bytes32 public constant PERMIT_TYPEHASH = keccak256(
    "Permit(address owner,address spender,uint256 value)"
    // Missing: uint256 nonce, uint256 deadline
);

function permit(address owner, address spender, uint256 value, bytes calldata sig) external {
    bytes32 structHash = keccak256(abi.encode(PERMIT_TYPEHASH, owner, spender, value));
    bytes32 digest = keccak256(abi.encodePacked("\x19\x01", DOMAIN_SEPARATOR, structHash));
    require(ECDSA.recover(digest, sig) == owner, "Invalid sig");
    // No nonce check → unlimited replay
    // No deadline → permanent validity
    // No chainId in domain → cross-chain replay
    _approve(owner, spender, value);
}
```

**Secure:**
```solidity
bytes32 public constant PERMIT_TYPEHASH = keccak256(
    "Permit(address owner,address spender,uint256 value,uint256 nonce,uint256 deadline)"
);

function permit(
    address owner, address spender, uint256 value,
    uint256 deadline, uint8 v, bytes32 r, bytes32 s
) external {
    require(block.timestamp <= deadline, "Expired");
    bytes32 structHash = keccak256(abi.encode(
        PERMIT_TYPEHASH, owner, spender, value, nonces[owner]++, deadline
    ));
    bytes32 digest = _hashTypedDataV4(structHash); // Includes chainId + address(this)
    require(ECDSA.recover(digest, v, r, s) == owner, "Invalid sig");
    _approve(owner, spender, value);
}
```

### 2.6 Copy-Paste Inheritance Bugs

LLMs copy patterns from one contract and paste into another, breaking overrides:

```solidity
// AI copies ERC721 transfer hook but uses wrong super call order
contract MyNFT is ERC721, ERC721Enumerable, Pausable {
    function _update(address to, uint256 tokenId, address auth)
        internal override(ERC721, ERC721Enumerable) returns (address)
    {
        // Missing whenNotPaused check copied from example
        return super._update(to, tokenId, auth);
        // Also missing: ERC721Enumerable._update must be called via super
        // In diamond inheritance, super chain matters — AI often breaks it
    }
}
```

## 3. Audit Checklist for AI-Assisted Contracts

Run this in addition to the standard Phase 3 checklist when AI generation is suspected.

### 3.1 Pre-Review

- [ ] Search git log for mass commits (>20 files, uniform structure)
- [ ] Check for "generated by", "copilot", or "AI" comments in source
- [ ] Verify OZ version consistency across all imports
- [ ] Identify any functions with no NatSpec (AI sometimes omits for "obvious" functions)

### 3.2 Per External Function

- [ ] CEI order: Checks → Effects → Interactions (no exceptions)
- [ ] Reentrancy: `nonReentrant` on every external function that calls out
- [ ] Access control: check modifier exists AND covers all edge cases (init, proxy, admin)
- [ ] Interface calls: verify function signature against actual deployed interface
- [ ] Error messages: `require()` statements have descriptive messages

### 3.3 Per Signature Validation

- [ ] Domain separator includes `chainId` and `address(this)`
- [ ] TypeHash includes `nonce` and `deadline`
- [ ] Nonce is incremented after use (not reusable)
- [ ] Deadline is enforced before signature recovery

### 3.4 Per Upgradeable Contract

- [ ] `constructor() { _disableInitializers(); }` present
- [ ] `initialize()` has `initializer` modifier (not `onlyOwner`)
- [ ] All state variables use namespaced storage (ERC-7201) or are in correct upgrade-safe slots
- [ ] `_authorizeUpgrade()` has proper access control

## 1. EIP-7732 — Enshrined Proposer-Builder Separation (ePBS)

### 1.1 Architecture Change

**Before ePBS (current):** PBS is off-protocol via MEV-Boost relays.
Proposers outsource block building to builders via a trusted relay (Flashbots, etc.).
The relay is a single point of failure and trust.

```
Proposer ← relay (trusted) ← Builder
              ↑
         off-chain PBS (current)
```

**After ePBS (EIP-7732):** PBS is enshrined at consensus level.
Two distinct roles with on-chain commitments:
1. **Proposer** — commits to a `SignedExecutionPayloadHeader` (block body hash + value)
2. **Payload Provider (Builder)** — reveals the actual `ExecutionPayload`

```
Slot N:   Proposer commits to header  ─── on-chain
Slot N+1: Builder reveals payload    ─── on-chain
          (or payload withheld → empty slot)
```

### 1.2 New Attacker Models

#### 1.2.1 Payload Withholding Attack

The builder commits to a value but withholds the payload, causing an empty slot.
Motivation: griefing a specific protocol's time-sensitive operation (liquidations,
oracle updates, keeper calls).

**Smart contract implications:**
- Contracts that assume consistent block production can be bricked for 1 slot
- Liquidation protocols: a withheld slot delays liquidations → bad debt accumulation
- Keeper networks: missed upkeep windows if exactly 1 slot withheld
- Time-sensitive auctions or claims with tight deadlines

**Audit check:**
- [ ] Does the protocol have off-by-one-slot tolerance for time-sensitive operations?
- [ ] Are liquidation thresholds calculated with 1-slot withholding grace?
- [ ] Do keeper rewards account for missed slots due to payload withholding?

#### 1.2.2 Builder Front-Running (Formalized MEV)

With ePBS, builders have a formal, on-chain role. A malicious builder can:
- Include their own transactions before and after a victim's (sandwich)
- Selectively exclude transactions from specific protocols
- Reorder transactions to maximize extraction with no relay accountability

The difference from current MEV: with relays, there's social accountability.
With ePBS, builders are anonymous on-chain roles with no reputation requirement.

**Audit check:**
- [ ] Are slippage protections enforced on-chain (not just frontend)?
- [ ] Do AMM swaps use `amountOutMinimum` / `sqrtPriceLimitX96`?
- [ ] Does the protocol assume any ordering guarantees within a block?

#### 1.2.3 Preconfirmation Timing Attacks

ePBS enables "preconfirmations" (slot N proposer commits to include TX in slot N).
Contracts that rely on preconf guarantees before they are final are vulnerable.

```solidity
// Vulnerable: assumes preconf = finality
function settleOnPreconf(bytes32 txHash) external {
    require(preconfBridge.isConfirmed(txHash), "Not preconfirmed");
    _settle(); // Premature settlement — preconf can still be withheld
}
```

**Secure pattern:**
```solidity
function settle(bytes32 txHash) external {
    // Only settle on finalized inclusion (2 slots after payload reveal)
    require(
        block.number >= inclusionBlock[txHash] + FINALIZATION_DELAY,
        "Not finalized"
    );
    _settle();
}
```

### 1.3 Audit Checks for ePBS

- [ ] Does the protocol have any timing assumptions about slot production?
- [ ] Are liquidations, keepers, or oracle updates robust to 1–2 missed slots?
- [ ] Does the protocol trust preconfirmations before on-chain finalization?
- [ ] Are slippage protections enforced on-chain for all swaps?

## 2. EIP-7928 — Block Access Lists (BALs)

### 2.1 What Are BALs

Block Access Lists allow each transaction to declare, upfront, which accounts and
storage slots it will read and write. Builders include this declaration in the block.

```
BlockBody {
    transactions: [TX1, TX2, ...],
    accessLists: [         ← new: parallel array
        BAL_for_TX1,
        BAL_for_TX2,
        ...
    ]
}
```

This is analogous to EIP-2930 (per-TX access lists for gas discounts) but at the
block level, declared by builders for parallelization and transparency.

### 2.2 Security Implications

#### 2.2.1 MEV Transparency — New Frontrunning Surface

Currently MEV bots hide their state access patterns to prevent being copied.
With BALs, every transaction's access pattern is declared upfront.

**Implications:**
- Searchers can see exactly which storage slots a pending TX will touch
- A frontrunner can copy-sandwich with perfect knowledge of state touched
- Privacy-preserving mechanisms (commit-reveal, encrypted mempools) become
  more important for protocols relying on ordering opacity

**Audit check:**
- [ ] Does the protocol rely on ordering opacity or "secret" state reads for security?
- [ ] Are MEV-sensitive operations (e.g., arbitrage, liquidations) using private mempools?

#### 2.2.2 Access List Lying (Underdeclared Reads/Writes)

If a TX touches storage slots not declared in its BAL, behavior depends on implementation:
- Some clients may reject underdeclared TXes at the builder level
- A mismatch between declared and actual access sets could allow selective exclusion

**Audit check:**
- [ ] If the protocol submits transactions with access lists, are they complete and correct?
- [ ] Does the protocol validate that on-chain access lists match expected state access?

#### 2.2.3 Parallelization Assumptions

BALs enable parallel TX execution (non-conflicting access sets run concurrently).
Contracts that assume strict sequential execution within a block may break.

```solidity
// Vulnerable assumption: TX1 runs before TX2 in same block
// With parallelization, TX1 and TX2 may execute in undefined order
// if their access sets don't conflict

contract PriceOracle {
    uint256 public price; // Written by TX1
    uint256 public lastUpdate; // Written by TX1

    function getPrice() external view returns (uint256) {
        // TX2 may read stale price if TX1 runs concurrently
        require(block.timestamp - lastUpdate < MAX_STALENESS);
        return price;
    }
}
```

**Secure pattern:** Use atomic packed storage for values that must be consistent:
```solidity
// Pack price + timestamp in one slot — atomic read/write
struct PriceData {
    uint128 price;
    uint128 timestamp;
}
mapping(address => PriceData) public priceData; // Single slot = single access unit
```

### 2.3 Audit Checks for BALs

- [ ] Does the protocol assume strict sequential execution within a block?
- [ ] Are price + timestamp values in separate storage slots (non-atomic)?
- [ ] Does the protocol rely on MEV opacity for security guarantees?
- [ ] If the contract submits access-listed TXes, are the lists correct and complete?

## 10. dYdX v4 — Cosmos App-Chain Risks

dYdX v4 runs on its own Cosmos SDK chain with an off-chain order book and
on-chain settlement. This introduces risks absent from EVM-native perp DEXes.

### 10.1 Off-Chain Order Book Trust Assumptions

Validators maintain the order book off-chain. A malicious validator set can:
- Reorder fills to frontrun large orders
- Selectively delay order inclusion (MEV without on-chain evidence)
- Collude to fill orders at stale prices before oracle updates

**Audit check:**
- [ ] Are fills price-validated on-chain against the oracle at execution time?
- [ ] Is there a maximum acceptable spread between fill price and oracle price?
- [ ] Are order timestamps validated to prevent stale order execution?

### 10.2 Cosmos Validator Set Compromise

Unlike Ethereum mainnet, dYdX v4 security depends on dYdX's own validator set.
A 33%+ Byzantine fault can halt the chain; a 67%+ fault can equivocate.

**Smart contract / bridge implications:**
- Funds bridged from Ethereum to dYdX v4 depend on the IBC bridge validators
- A chain halt freezes all user funds until validators recover
- No equivalent of Ethereum's social consensus for fork resolution

**Audit check:**
- [ ] Is the bridge IBC-native (light client) or multisig-based?
- [ ] What is the validator set decentralization? (number of validators, stake distribution)
- [ ] Are there emergency withdrawal mechanisms that don't require chain liveness?

### 10.3 Sequencer / Mempool Manipulation

dYdX v4 uses CometBFT consensus. Block proposers can reorder transactions
within their proposal window, similar to Ethereum block proposers.

```go
// Conceptual: validator reorders orders to sandwich
// Order1 (buy 100 ETH at market) → validator inserts own buy before → price up
// Order1 fills at worse price → validator sells → profit
```

**Audit check:**
- [ ] Does the protocol use TWAP-based fills to mitigate single-block price manipulation?
- [ ] Are large orders split or time-weighted to reduce sandwiching profit?

## 11. Gains Network (gTrade) — Synthetic Asset Risks

gTrade uses a single DAI vault as the counterparty for all synthetic perps.
No real underlying assets are held — only DAI collateral.

### 11.1 Vault Solvency (DAI Vault as Counterparty)

The DAI vault pays out winning traders and receives from losers.
If the vault is underfunded relative to outstanding exposure, it cannot pay winners.

**Vulnerable scenario:**
```
Vault: 10M DAI
Open longs: 5M DAI notional (all profitable at current price)
Vault PnL liability: 8M DAI (exceeds vault if leverage > 1.6x average)
```

**Audit check:**
- [ ] Is there a maximum open interest cap relative to vault size?
- [ ] How is vault utilization enforced (on open, on PnL accumulation, or both)?
- [ ] What happens when vault drops below minimum solvency? (paused trading? forced deleveraging?)

### 11.2 Oracle Dependency on Single Source

gTrade uses Chainlink + custom price aggregation. A single feed failure
or manipulation affects all synthetic markets simultaneously.

**Audit check:**
- [ ] Is there a price deviation circuit breaker across feeds?
- [ ] Can a single Chainlink feed being stale halt the entire protocol?
- [ ] Is there a fallback oracle for each market?

### 11.3 Collateral Factor Manipulation

Since there's no underlying, collateral factors are parameters.
A governance attack that inflates collateral factors allows undercollateralized positions.

**Audit check:**
- [ ] Are collateral factor changes timelocked?
- [ ] Is there a maximum single-governance-action change to collateral factors?

## 12. Advanced Funding Rate Manipulation

### 12.1 Skew-Based Funding Manipulation

Protocols that set funding rate proportional to long/short skew can be manipulated
by a whale who dominates one side to extract funding from the other.

**Attack flow:**
1. Attacker opens massive long (dominates long side → funding rate goes to max positive)
2. Short holders pay max funding to attacker each hour
3. Attacker's position profit > funding cost if oracle price is stable

**Mitigations to check:**
- [ ] Is there a maximum position size per address or per block?
- [ ] Is funding capped at a maximum rate regardless of skew?
- [ ] Is there a delay between skew change and funding rate update (prevents flash manipulation)?

### 12.2 Time-Weighted Funding Gaming

If funding is calculated as a time-weighted average, attackers can:
- Open large skewing positions just before funding snapshot
- Close positions immediately after snapshot
- Cost: only hold for the snapshot duration (1 block or 1 slot)

**Audit check:**
- [ ] Is the funding rate computed on a TWAP basis (minutes/hours, not 1 block)?
- [ ] Are there minimum hold times before receiving accumulated funding?

## 13. Insurance Fund Attacks

### 13.1 Insurance Fund Drain via Cascading Liquidations

The insurance fund absorbs bad debt when liquidations fail (position underwater
before liquidators act). A coordinated attack can drain it:

1. Attacker opens max-leverage long in an illiquid market
2. Attacker (with another wallet) dumps the asset rapidly
3. Liquidators cannot close fast enough → position goes negative
4. Insurance fund absorbs bad debt

**Audit check:**
- [ ] Is open interest in illiquid markets capped?
- [ ] Is the maximum leverage reduced for illiquid markets?
- [ ] What is the maximum single-position bad debt vs. insurance fund size?

### 13.2 Insurance Fund → Socialized Loss Timing

When the insurance fund is exhausted, some protocols socialize losses across all
open positions (reduce PnL for winners). A griefing attack:

1. Drain insurance fund via cascading liquidations
2. Wait for a winning trader to have a large unrealized profit
3. Trigger socialized loss → the winning trader's PnL is clawed back

**Audit check:**
- [ ] Is the socialization mechanism transparent and predictable?
- [ ] Are there caps on how much can be socialized per epoch?
- [ ] Can socialized loss be triggered by an attacker (is it permissionless)?

## 14. Cross-Margin vs. Isolated Margin

### 14.1 Cross-Margin Contagion

In cross-margin mode, a loss in one market reduces margin available for all other
markets, triggering cascading liquidations across multiple positions.

**Vulnerable pattern:**
```solidity
function getAccountHealth(address trader) public view returns (int256) {
    int256 totalPnL = 0;
    for (uint256 i = 0; i < markets.length; i++) {
        totalPnL += getUnrealizedPnL(trader, markets[i]); // All markets share margin
    }
    return collateral[trader] + totalPnL;
}
```

**Attack:** Open a position in market A to be profitable, cross-margin subsidizes
a large leveraged position in market B. Manipulate market B price to trigger
liquidation → market A profits are seized to cover market B losses.

**Audit check:**
- [ ] Is there a maximum cross-margin exposure ratio per market?
- [ ] Can cross-margin mode be disabled per market (to prevent contagion)?

### 14.2 Isolated Margin Bypass

Some protocols allow switching between cross and isolated margin. An attacker
can switch from isolated to cross margin immediately before a bad debt event
to socialize the loss:

```solidity
// Attacker: position about to be liquidated (isolated)
// Switch to cross-margin → position's bad debt becomes cross-margin bad debt
// socialized across all other cross-margin users
isolatedToChross(positionId); // If no timelock → instant socialization of bad debt
```

**Audit check:**
- [ ] Is there a timelock between switching from isolated to cross-margin?
- [ ] Can margin mode be switched on a position that is below the maintenance margin?
```

## 7. Noir Language Security

Noir is an SNARK-friendly DSL for writing ZK circuits that compiles to ACIR
(Abstract Circuit Intermediate Representation). Key security considerations:

### 7.1 `unconstrained` Functions

`unconstrained` functions execute off-circuit as witness generation hints.
They are NOT constrained — their outputs must be verified by constrained code.

**Vulnerable — unconstrained result used directly:**
```rust
// Noir
unconstrained fn compute_sqrt(x: Field) -> Field {
    // This runs off-circuit — result is NOT verified to be correct sqrt
    std::field::sqrt(x)
}

fn verify_sqrt(x: Field, result: Field) {
    let hint = compute_sqrt(x); // hint is unconstrained
    // MISSING: constrained check that result^2 == x
    assert(hint == result); // Only checks hint == result, not correctness
}
```

**Secure — constrained verification of unconstrained hint:**
```rust
fn verify_sqrt(x: Field, result: Field) {
    let hint = compute_sqrt(x); // Off-circuit hint
    assert(hint * hint == x);   // On-circuit constraint: result^2 == x
    assert(hint == result);
}
```

**Audit check:** Every `unconstrained` function's output must be verified by
a constrained assertion before being used in business logic.

### 7.2 Public vs. Private Input Confusion

Noir distinguishes `pub` (public) from private inputs. Public inputs are
visible to verifiers; private inputs are not. Misclassification leaks secrets
or allows input substitution.

**Vulnerable:**
```rust
// Secret should be private, but declared pub — leaks to verifier
fn main(pub secret_key: Field, pub message_hash: Field) -> pub Field {
    // secret_key is now visible in the proof's public inputs
    pedersen_hash([secret_key, message_hash])
}
```

**Secure:**
```rust
fn main(secret_key: Field, pub message_hash: Field) -> pub Field {
    // secret_key is private (default) — not visible in public inputs
    pedersen_hash([secret_key, message_hash])
}
```

**Audit check:**
- [ ] Are all secret values (private keys, salts, hidden amounts) declared without `pub`?
- [ ] Are all values needed for on-chain verification declared as `pub`?

### 7.3 Field Arithmetic Overflow

Noir uses BN254 scalar field by default (p ≈ 2^254). Values that appear to be
safe `uint256` may overflow the field prime.

**Audit check:**
- [ ] Are range checks applied to inputs that must fit in 128 or 64 bits?
- [ ] Are comparisons in Noir using field arithmetic or bit-constrained arithmetic?

## 8. SP1 (Succinct) — Deep Dive

SP1 is a performant zkVM that proves execution of RISC-V programs (written in Rust).
It uses STARKs internally and wraps in a Groth16 or PLONK proof for on-chain verification.

### 8.1 Precompile Security

SP1 provides precompiles for expensive operations (SHA-256, Keccak, secp256k1, etc.).
Using precompiles incorrectly leads to constraint mismatches.

**Audit check:**
- [ ] Are precompile calls (e.g., `sp1_zkvm::precompiles::sha256::compress`) used
  only for their exact intended inputs?
- [ ] Is the precompile output validated against expected on-chain computation?

### 8.2 Cycle Limits and Denial of Service

SP1 proofs have a maximum cycle count. A user-supplied input that causes excessive
cycles can make proofs unprovable (DoS on the prover).

**Vulnerable:**
```rust
// Guest program: unbounded loop based on user input
fn main() {
    let n = sp1_zkvm::io::read::<u64>(); // User-controlled
    for i in 0..n { // n = u64::MAX → cycles out
        // ... expensive computation
    }
}
```

**Secure:**
```rust
fn main() {
    let n = sp1_zkvm::io::read::<u64>();
    assert!(n <= MAX_ITERATIONS, "Input too large");
    for i in 0..n { ... }
}
```

**Audit check:**
- [ ] Are all user-controlled loop bounds capped to a safe maximum?
- [ ] Is the maximum cycle count validated in CI against the on-chain proof budget?

### 8.3 Committed Input / Output Integrity

SP1 uses `sp1_zkvm::io::commit()` to write values to the public output (journal).
The on-chain verifier must check ALL committed outputs.

**Audit check:**
- [ ] Are all security-relevant values committed (not just the final result)?
- [ ] Does the on-chain verifier decode and validate EVERY committed value?
- [ ] Is there a nonce or block number committed to prevent proof replay?

## 9. Polygon CDK — Custom Chain Deployment

Polygon CDK allows deploying custom ZK-rollup chains using Polygon's proving stack.
Each CDK chain has its own contracts on L1 and a centralized sequencer by default.

### 9.1 Sequencer Centralization

CDK chains ship with a centralized sequencer. Unlike zkSync Era or Starknet,
there's no decentralized sequencer rotation by default.

**Audit check:**
- [ ] Who controls the sequencer? Is there a fallback (force inclusion) mechanism?
- [ ] Can the sequencer censor transactions indefinitely?
- [ ] Is there a force-include timelock for users to bypass the sequencer?

### 9.2 Bridge Security (LxLy)

CDK uses the LxLy bridge for L1↔L2 communication. The bridge is shared across
multiple CDK chains — a bug affects all chains using the same L1 bridge contract.

**Audit check:**
- [ ] Is the LxLy bridge being used for arbitrary message passing (not just tokens)?
- [ ] Are message origins properly validated on the destination chain?
- [ ] Is the bridge contract audited and immutable, or upgradeable?

### 9.3 Proof Aggregation Risks

CDK chains can batch proofs on L1. If aggregation logic is incorrect,
a valid proof for chain A could be replayed for chain B.

**Audit check:**
- [ ] Does each CDK chain's proof include the `chainId` as a public input?
- [ ] Is the verifier contract chain-specific (separate deployment per chain)?

## 10. Folding Schemes (Nova, SuperNova, ProtoStar)

Folding schemes enable Incrementally Verifiable Computation (IVC): proving a
long sequential computation by "folding" steps together instead of proving each step.

### 10.1 What They Are

Instead of one monolithic circuit, the prover accumulates ("folds") computation
steps incrementally. Only the final folded instance needs a succinct proof.

Used in: recursive zkVMs (proving many EVM steps), client-side proving, rollup aggregation.

### 10.2 Security Considerations

**Cross-step state consistency:** Each step's output must equal the next step's input.
An incomplete constraint between steps allows state injection.

**Audit check for IVC-based systems:**
- [ ] Is the "running instance" (accumulated state) correctly constrained between steps?
- [ ] Are there checks that the initial state matches the known genesis state?
- [ ] If the folding scheme has a trusted setup, was it conducted correctly?

**Soundness of the accumulator:** The accumulator must not allow a prover to
substitute a fake intermediate step without the verifier detecting it.

```
Step1 → Step2 → Step3 → ... → StepN
  ↕fold  ↕fold  ↕fold       ↕final proof

// If StepK output != StepK+1 input, the proof should fail
// Incorrect fold constraints allow "teleporting" state across steps
```

**Audit check:**
- [ ] Is the folding verifier from an audited library (e.g., Nova by Microsoft Research,
  folding-schemes by Lita Foundation)?
- [ ] Are there independent audits of the folding scheme's soundness?
```

## xUSD / Stream Finance ($285M, November 2025)

**Vulnerability**: Hardcoded Oracle Price + Recursive Leverage Collapse
**Category**: Oracle Price Hardcoding (§4.7), CeDeFi Recursive Leverage (§4.7)
**Loss**: ~$285M
**Date**: November 2025

### Background

xUSD was a "CeDeFi" stablecoin protocol that allowed users to mint xUSD by
depositing various collateral assets. The protocol used a custom oracle adapter
for its primary collateral token (xAsset). The adapter returned a hardcoded price
of `$1.00` for xAsset regardless of its actual market price.

### Vulnerable Code

```solidity
contract xAssetPriceAdapter {
    // "Price" is always $1.00 — never fetches real market price
    function getQuote(
        uint256 inAmount,
        address base,
        address quote
    ) external pure returns (uint256 outAmount) {
        // Hardcoded: 1 xAsset = 1 USD (1e18 in 18-decimal representation)
        // No Chainlink call, no TWAP, no validation
        return inAmount; // 1:1 peg assumption baked in forever
    }
}
```

### Attack Flow

1. **Setup**: xAsset trades at $0.80 on the open market due to protocol stress.
   The oracle still returns `$1.00`.

2. **Initial deposit**: Attacker deposits 1M xAsset (worth $800K market).
   Protocol prices it at $1M (hardcoded). Attacker mints $800K xUSD (80% LTV).

3. **Recursive leverage**: Attacker swaps xUSD for more xAsset on DEX
   (market price $0.80), deposits again. Each loop amplifies the overcollateralization
   illusion while building up real exposure to xAsset's declining price.

4. **Leverage loop**: After 10 iterations, attacker controls $8M nominal collateral
   (real value: $6.4M), with $6M xUSD minted.

5. **Collapse**: When xAsset price drops further, the protocol is undercollateralized
   across ALL users who used the same adapter. The oracle is still returning `$1.00`.
   Total bad debt: $285M across protocol.

### Why Static Analysis Missed It

The oracle adapter was a separate contract with a clean interface. Slither
detected no issues — the function was `pure` (no storage reads, no external calls).
The hardcoded price was a design decision that looked like a simplification.

### Secure Pattern

```solidity
contract xAssetPriceAdapter {
    AggregatorV3Interface public immutable chainlinkFeed;
    uint256 public constant MAX_STALENESS = 1 hours;
    uint256 public constant MAX_DEVIATION_BPS = 500; // 5% max deviation from peg

    function getQuote(
        uint256 inAmount,
        address base,
        address quote
    ) external view returns (uint256 outAmount) {
        // Always fetch real price
        (,int256 price,, uint256 updatedAt,) = chainlinkFeed.latestRoundData();
        require(block.timestamp - updatedAt <= MAX_STALENESS, "Stale price");
        require(price > 0, "Invalid price");

        uint256 realPrice = uint256(price); // 1e8 decimals from Chainlink

        // Circuit breaker: if price deviates >5% from peg, halt minting
        uint256 pegPrice = 1e8; // $1.00 in Chainlink decimals
        uint256 deviation = realPrice > pegPrice
            ? ((realPrice - pegPrice) * 10_000) / pegPrice
            : ((pegPrice - realPrice) * 10_000) / pegPrice;
        require(deviation <= MAX_DEVIATION_BPS, "Price too far from peg");

        return (inAmount * realPrice) / pegPrice;
    }
}
```

### Key Lessons

- **Audit all oracle adapters, not just the oracle itself.** The Chainlink feed was fine;
  the adapter silently discarded it.
- **`pure` functions can be catastrophically wrong.** Static correctness ≠ economic correctness.
- **Recursive leverage is a multiplier for oracle errors.** A 10% oracle discrepancy
  becomes 100% bad debt at 10x leverage.
- **ERC-7726 adapters need explicit validation.** See `vulnerability-taxonomy.md §4.5`.

## Hyperliquid HLP Structural Manipulation (2025)

**Vulnerability**: Vault as Liquidation Absorber + Oracle Centralization
**Category**: Perpetual DEX Design (perpetual-dex.md §9)
**Loss**: ~$200M (potential exposure, partially realized)
**Date**: Multiple incidents, 2025

### Background

Hyperliquid is a high-performance on-chain perpetuals DEX. Its HLP (Hyperliquidity
Provider) vault acts as the market maker and—critically—as the default liquidation
absorber for bad debt. When liquidations produce bad debt that the insurance fund
can't cover, HLP absorbs it.

### The Structural Vulnerability

```
┌─────────────────────────────────────────────────────┐
│                    HLP Vault                        │
│  - Main market maker (~$400M TVL)                   │
│  - Earns trading fees                               │
│  - ALSO absorbs all bad debt after insurance fund   │
└──────────────────────┬──────────────────────────────┘
                       │ Bad debt flows here
          ┌────────────┴──────────────┐
          │                           │
  Insurance Fund               Direct Absorption
  (small buffer)            (when fund exhausted)
```

The HLP vault's dual role as market maker + bad debt absorber creates a structural
conflict: the entity most harmed by bad debt is also the liquidity provider that
must continue operating through a crisis.

### Attack Flow (Reconstructed)

1. **Market selection**: Attacker identifies a low-liquidity market (e.g., JELLY token)
   where HLP has significant market-making positions.

2. **Leverage amplification**: Attacker takes a large leveraged long in the target market,
   intentionally exceeding their ability to cover losses.

3. **Price manipulation**: Attacker (with external liquidity) pumps the target asset's
   price on CEXes. Hyperliquid's oracle follows CEX prices with a short lag.

4. **Oracle gap exploitation**: During the lag, attacker's long is deeply profitable
   on Hyperliquid (oracle hasn't caught up). Attacker closes the long for profit.

5. **Bad debt generation**: Other market participants were short; HLP was short as
   market maker. When HLP liquidates its short, the insurance fund is exhausted.
   HLP absorbs the remaining bad debt directly — HLP depositors lose value.

### Oracle Centralization Risk

Hyperliquid uses a proprietary oracle run by Hyperliquid validators. This is:
- Not Chainlink (no SLA, no cryptographic attestation)
- Controlled by the same entity that runs the exchange
- Not transparent about its exact data sources or aggregation methodology

**Audit check for any protocol using a proprietary oracle:**
- [ ] Is the oracle source code open and auditable?
- [ ] How many independent validators contribute to the oracle?
- [ ] Is there a circuit breaker if the oracle deviates from external price sources?
- [ ] Who can update the oracle contract?

### Secure Vault Design Principles

```solidity
// Anti-pattern: vault is BOTH market maker AND bad debt absorber
contract HLPVault {
    // Vault deposits
    // Vault earns fees as market maker
    // Vault.balance -= badDebt; // DANGEROUS: depositors didn't sign up for bad debt
}

// Secure: separate concerns
contract LiquidityVault {
    // ONLY for earning market-making fees
    // Zero exposure to bad debt
}

contract InsuranceFund {
    // Separate pool for bad debt absorption
    // Clearly communicated risk to depositors
    uint256 public maxBadDebtPerEpoch; // Capped exposure
}
```

### Key Lessons

- **Vaults with dual roles (market maker + backstop) have hidden tail risk.**
  Depositors may not understand they're also the liquidation absorber.
- **Proprietary oracles are a single point of failure.** Even a 5-second lag
  is exploitable at scale.
- **Low-liquidity markets in a high-leverage system are attack vectors.**
  The cost of pumping a low-cap asset may be far less than the HLP exposure.
- **See `perpetual-dex.md §9`** for the detailed checklist.
```
